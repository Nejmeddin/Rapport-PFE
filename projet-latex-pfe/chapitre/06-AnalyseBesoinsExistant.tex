\section{Introduction}
Ce chapitre présente l'existant sur lequel s'appuie et s'intègre le projet. Il se structure en deux volets complémentaires. Le premier analyse Apache Solr, déjà opérationnel dans l'environnement SAP Commerce de DECADE, en identifiant ses limites directement liées à la qualité des données produit et en montrant comment l'enrichissement LLM les adresse. Le second évalue expérimentalement les solutions LLM disponibles (locales, gratuites, GPU cloud, payantes) afin de retenir le modèle le mieux adapté au pipeline d'enrichissement des fiches produit.

\section{Volet 1 — Apache Solr dans SAP Commerce (Existant interne)}

\subsection{Rôle et intégration}
Apache Solr est le moteur de recherche et d'indexation déjà intégré nativement dans l'environnement SAP Commerce Cloud (Hybris) de DECADE. Basé sur Apache Lucene, il assure trois fonctions critiques pour le catalogue e-commerce~: la recherche produit full-text, les filtres de navigation par facettes, et l'auto-complétion.

Son rôle dans l'architecture est complémentaire à la base de données Hybris~: là où Hybris stocke la source de vérité (les données produit), Solr maintient un index dédié, optimisé pour répondre aux requêtes utilisateurs en quelques millisecondes.

\subsection{Flux d'indexation}
Le processus d'indexation se déroule en cinq étapes séquentielles~:
\begin{enumerate}
  \item Les données produits sont stockées dans la base SAP Commerce~;
  \item Un processus d'indexation les extrait et les transforme en documents Solr~;
  \item Ces documents sont persistés dans l'index Solr~;
  \item Lorsqu'un utilisateur effectue une recherche, la requête est transmise à Solr~;
  \item Solr retourne les résultats avec les facettes associées.
\end{enumerate}

\subsection{Limitations structurelles de Solr}
Bien qu'opérationnel, Solr présente six limitations qui dégradent directement la qualité de recherche et motivent le présent projet~:

\subsubsection{Limitation 1 — Modèle lexical et dépendance aux mots-clés}
Solr analyse les tokens (mots) présents dans la requête et les rapproche des mots présents dans l'index. Il ne comprend pas la signification ou l'intention derrière la requête.

\begin{quote}
\textit{Exemple~: Requête \og best vacuum for pet hair \fg{} -> Solr analyse [best][vacuum][pet][hair] -> ne comprend pas \og aspirateur adapté aux poils d'animaux \fg{}.}
\end{quote}

\subsubsection{Limitation 2 — Absence de compréhension sémantique}
Solr ne possède pas de mécanisme natif de recherche sémantique. Il ne peut ni décoder l'intention utilisateur, ni gérer les synonymes implicites, ni classer les résultats par pertinence de sens.

\subsubsection{Limitation 3 — Personnalisation du ranking limitée}
En configuration standard, Solr retourne les mêmes résultats pour tous les utilisateurs effectuant la même requête. Les moteurs modernes s'appuient sur le machine learning pour personnaliser le ranking (historique de navigation, préférences). Solr exige des règles manuelles de boosting.

\subsubsection{Limitation 4 — Complexité de configuration et maintenance}
Améliorer la pertinence nécessite des interventions techniques importantes~: règles de boosting, dictionnaires de synonymes, analyseurs linguistiques. Cette complexité accroît les coûts de maintenance et exige une expertise pointue.

\subsubsection{Limitation 5 — Dépendance à l'indexation complète}
Lors d'un Full Index, l'index existant est remplacé, ce qui peut créer une indisponibilité temporaire de la recherche. Sur un catalogue de grande taille, l'opération est longue et coûteuse en ressources.

\subsubsection{Limitation 6 — Qualité des résultats tributaire de la qualité des données}
C'est la limitation la plus directement liée au projet~: si les attributs produit sont incomplets ou hétérogènes dans Hybris, l'index Solr reproduit ces lacunes. Les facettes (couleur, matière, saisonnalité) ne peuvent exister que si les attributs correspondants sont renseignés dans la source.

\subsection{Illustration concrète — Cas Jacadi}
Un exemple réel illustre ces limitations~: dans le catalogue Jacadi (vêtements enfants), la recherche du terme \og table \fg{} retourne des produits vestimentaires. Plusieurs mécanismes combinés l'expliquent~:

\begin{itemize}
  \item \textbf{Free Text Search}~: Solr indexe la description complète des produits. Une description comme \og idéal pour dîner à table en famille \fg{} contient le token \og table \fg{}~;
  \item \textbf{Fuzzy Search}~: la recherche approximative peut matcher \og table \fg{} sur \og tableau \fg{} ou \og tablette \fg{}~;
  \item \textbf{Synonymes configurés}~: si le backoffice définit \og table $\leftrightarrow$ furniture \fg{}, la requête s'étend à \og table OR furniture \fg{}~;
  \item \textbf{Tokenisation}~: un analyseur mal configuré peut extraire \og table \fg{} depuis \og comfortable \fg{}.
\end{itemize}

\subsection{En quoi l'enrichissement LLM améliore Solr}
L'enrichissement LLM ne remplace pas Solr, il le complète en alimentant l'index avec des attributs structurés, normalisés et fiables.

\subsubsection{Amélioration des facettes}
En générant des attributs enrichis (couleur, matière, occasion, style, saisonnalité), le LLM crée les champs manquants qui deviendront des facettes exploitables dans Solr.

\section{Volet 2 — Évaluation des solutions LLM disponibles}

\subsection{Objectif et protocole}
Avant de retenir un modèle pour la production, une phase d'exploration comparative a été menée. L'objectif est de qualifier chaque approche sur trois critères~: capacité à traiter des scénarios réalistes, stabilité, et viabilité pour un usage productif continu.

Les tests ont été conduits sur quatre niveaux de complexité croissante~:
\begin{enumerate}
  \item \textbf{Texte seul}~: description produit sans image~;
  \item \textbf{Image unique}~: une seule image produit encodée en base64~;
  \item \textbf{Multi-images}~: 3 à 4 vues du même produit~;
  \item \textbf{Images + métadonnées}~: multi-images + attributs existants + description textuelle.
\end{enumerate}

Ce dernier scénario représente le cas d'usage réel du projet et génère un contexte volumétrique élevé.

\subsection{Axe 1 — Exécution locale via Ollama}

\subsubsection{Présentation}
Ollama est un outil open source permettant d'exécuter des LLMs directement en local, sans API externe. Ses avantages théoriques sont la confidentialité des données et l'absence de quota. Trois modèles multimodaux ont été testés~: Gemma 4 (Google DeepMind), Qwen-VL (Alibaba) et Moondream (open source).

\subsubsection{Résultats}
L'exécution locale révèle une limite structurelle matérielle~: sans GPU dédié, le traitement repose entièrement sur le CPU, causant une latence prohibitive, une saturation de la RAM par les images encodées en base64, et des instabilités répétées sur les cas multi-images.

\subsubsection{Verdict}
\textbf{Non viable pour la production.} Exploitable uniquement pour exploration et validation de POC en dehors de toute contrainte de temps de réponse.

\subsection{Axe 2 — APIs gratuites (OpenRouter, Groq)}

\subsubsection{OpenRouter (Free Tier)}
OpenRouter est une plateforme d'agrégation d'APIs LLM. Le tier gratuit impose une fenêtre de contexte réduite.

\begin{itemize}
  \item \textbf{Gemma 4 (Google)}~: réussit sur texte et image unique~; context overflow sur multi-images et images + métadonnées~;
  \item \textbf{Nemotron Nano 12B 2 VL (NVIDIA)}~: réussit jusqu'au multi-images, mais context overflow sur le scénario complet~;
  \item \textbf{Qwen 3.6 Plus (Alibaba)}~: crash dès la première image~; erreurs HTTP 429 dès le premier appel.
\end{itemize}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\linewidth]{image_2.png}
  \caption{Erreur HTTP 429 sur OpenRouter — quota dépassé sur multi-images}
  \label{fig:openrouter-429}
\end{figure}

\subsubsection{Groq (Free Tier)}
Groq propose une inférence ultra-rapide via des puces LPU propriétaires. Le tier gratuit est limité à \textbf{12 000 tokens par minute (TPM)}.

Pour un seul produit avec 4 images et ses métadonnées, le test a consommé \textbf{46 214 tokens}, dépassant le quota d'un facteur 3,8$\times$. Le serveur retourne systématiquement~:

\begin{verbatim}
Error 413: Request Too Large for 'llama-3.3-70b-versatile'
Limit 12,000 TPM | Requested 46,471 tokens
-> rate_limit_exceeded
\end{verbatim}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\linewidth]{image_3.png}
  \caption{Dépassement du quota TPM Groq ($\times$3,8 sur un seul produit avec métadonnées)}
  \label{fig:groq-quota}
\end{figure}

\subsubsection{Verdict}
\textbf{Utile pour le prototypage rapide uniquement.} La contrainte de quotas rend ces plateformes incompatibles avec un flux d'enrichissement e-commerce soutenu.

\subsection{Axe 3 — GPU cloud (Google Colab)}

Pour contourner les limites matérielles locales, des tests ont été conduits sur Google Colab avec GPU gratuit. Deux modèles lourds ont été évalués~:

\subsubsection{Qwen 3.5 27B}
Latence~: \textbf{12 minutes} pour une phrase simple. Consommation GPU~: 13,3 Go sur 15,0 Go disponibles. Marge de sécurité insuffisante pour un traitement en lot.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\linewidth]{test_qwen3.5.27b_avec_colab.png}
  \caption{Qwen 3.5 27B sur Colab — latence de 12 min, RAM GPU à 13,3/15 Go}
  \label{fig:colab-qwen}
\end{figure}

\subsubsection{Gemma 31B}
Latence~: \textbf{5 minutes}. Consommation GPU~: 14,0 Go sur 15,0 Go, frôlant l'erreur Out Of Memory. RAM système à 11,5 Go.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\linewidth]{test_gemma31b_avec_colab.png}
  \caption{Gemma 31B sur Colab — saturation GPU critique à 14,0/15 Go}
  \label{fig:colab-gemma}
\end{figure}

\subsubsection{Verdict}
\textbf{Non viable pour la production.} La latence prohibitive et les marges d'erreur minimales rendent ces configurations inadaptées à un pipeline batch.

\subsection{Synthèse de décision}
L'étude expérimentale établit une trajectoire claire~:

\begin{itemize}
  \item \textbf{Local (Ollama)}~: exploration et POC, latence prohibitive sans GPU dédié~;
  \item \textbf{APIs gratuites (OpenRouter, Groq)}~: prototypage rapide, quotas incompatibles avec un flux continu~;
  \item \textbf{GPU cloud gratuit (Colab)}~: investigations techniques, latence et marges inacceptables~;
  \item \textbf{APIs payantes}~: seule option réaliste pour la robustesse et la continuité de service requises.
\end{itemize}

La stratégie retenue est \textbf{hybride}~: utiliser les offres gratuites pour le prototypage initial et l'ajustement des prompts, puis basculer vers une API payante pour la validation industrielle et le déploiement progressif.

\section{Conclusion}
Ce chapitre a établi l'état de l'existant sur deux axes. D'une part, Apache Solr — déjà opérationnel dans le projet — présente six limitations structurelles, dont la principale est la dépendance directe à la complétude des attributs produit dans Hybris~: des attributs manquants ou incohérents produisent un index Solr appauvri, des facettes peu fiables et une pertinence de recherche dégradée. L'enrichissement LLM adresse ce verrou en alimentant Hybris — et donc Solr — avec des attributs structurés, normalisés et validés. D'autre part, l'évaluation comparative des solutions LLM démontre que seules les APIs payantes offrent le niveau de robustesse requis pour industrialiser ce pipeline. Ces deux conclusions guident directement les choix de conception présentés dans le chapitre suivant.
